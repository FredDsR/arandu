Generate a conventional commit message for the current changes, stage them, and create the commit.

Steps:
1. Run `git status` to see all tracked/untracked changes.
2. Run `git diff --stat` and `git diff` to see unstaged changes.
3. Run `git diff --cached --stat` and `git diff --cached` to see already-staged changes.
4. Run `git log --oneline -5` to see recent commit style.
5. Analyze the changes and draft a commit message following Conventional Commits format: `<type>(<scope>): <description>`.
6. The message body should summarize the "why" with concise bullet points when multiple changes are involved. Each bullet point must be a single unwrapped line — do not insert line breaks within the same paragraph or bullet.
7. Show the commit message to the user and ask for confirmation before proceeding.
8. Stage the relevant files with `git add` (prefer adding specific files by name — avoid `git add -A` or `git add .`).
9. Create the commit using the generated message.
10. Confirm that the commit was created successfully.

Rules:
- Use types: feat, fix, docs, style, refactor, perf, test, build, ci, chore, revert
- Scope is optional but encouraged (e.g., versioning, cli, batch, qa)
- Keep the subject line under 72 characters
- Use imperative mood ("add feature" not "added feature")
- Do not commit files that likely contain secrets (.env, credentials, etc.)
- NEVER add a "Co-Authored-By" trailer or any self-referencing attribution (e.g., "Generated by...", "Created with...") to the commit message
- If the commit fails due to a pre-commit hook, fix the issue and create a NEW commit (do not amend)
